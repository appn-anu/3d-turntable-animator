# WebCodecs / mediabunny support matrix (Milestone 0 spike)

> Generated by the compat harness (`src/harness/compat.ts`) driven through
> Playwright (`tests/e2e/compat.spec.ts`) on this Linux machine.
> Reproduce with `npm run spike`. Engines: Playwright Chromium 1228, Firefox 1532.

## Verdict

The core bet is de-risked. A **streaming WebCodecs `VideoEncoder` -> mediabunny
`EncodedVideoPacketSource` -> MP4/WebM** pipeline produces playable files whose
timestamps, keyframes, decoder config (SPS/PPS), duration, and seek behaviour all
check out - in **both Chromium and Firefox**, at every target resolution.

On this machine H.264 encodes fine in Firefox too, so the VP9 -> WebM fallback was
not strictly required here - but it is built, wired to fall back automatically, and
independently confirmed to produce a playable WebM (see below). Software H.264
encode on other Linux setups is the real remaining unknown, which is exactly why
the fallback stays in.

## Full-pipeline spike (encode -> mux -> re-read -> play)

Preferred candidate = H.264 high on both engines. All four checks pass per config:
timestamps (index-derived microseconds), keyframes (frame 0 honoured), remux
(mediabunny re-reads its own output; codec string + SPS/PPS round-trip), and
playback (independent `<video>` decoder loads, sizes, and seeks). Encoding is in
software; the heaviest case (2048x2048@60, 60 frames) finishes in ~10 s.

Forced **VP9 -> WebM** at 1080x1080@30 also passes remux + playback on both engines
(`video/webm`, `vp09.00.40.08`), confirming the fallback is real, not just probeable.

## Key finding: decode-order emission (B-frames)

**Firefox's H.264 encoder emits packets in decode order** (reordered around
B-frames) even with `latencyMode: 'realtime'`, e.g. timestamps arrive
`[0, 66667, 33333, 133333, ...]`. Chromium emits in presentation order here.

Both are legal. The streaming muxer handles it correctly - packets are fed to
mediabunny in emission (decode) order carrying their true presentation timestamps,
and the output plays back fine. The consequence for our code:

- **Never assume emission order == frame order.** Timestamp verification compares
  the sorted *set*, and keyframe verification maps a packet's timestamp back to a
  source index rather than trusting emission position.
- Progress reporting in the render worker (Milestone 2) must count *submitted*
  frames, not emitted packets, since packets can lag and reorder.
